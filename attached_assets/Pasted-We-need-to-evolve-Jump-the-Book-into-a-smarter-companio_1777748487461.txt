We need to evolve Jump the Book into a smarter companion for real books people are actually reading now, not just public-domain demo books.

Core product direction:
A user is reading or listening to a modern book (for example sci-fi, fantasy, thriller, space opera, etc). They may not have an EPUB. They may be using Kindle or Audible. Since we cannot rely on official Kindle/Audible APIs or direct access to protected book files, the app should support a new workflow:

1. User enters a real book title and author
2. The app searches the public web for metadata and world context
3. The app builds a reusable “book bible” / “story profile”
4. The user confirms or edits the important details
5. The app uses that context plus the user’s reading position and optional excerpt/notes to generate a spoiler-safe visual companion
6. Generated scenes and images are stored and reused

This is not meant to be a piracy tool or an unofficial Kindle/Audible importer.
It is a companion experience builder using:
- user input
- public web metadata
- search-supported book context
- optional user-pasted excerpts
- generated original visuals

We need to implement this as a proper feature in the web app.

==================================================
MAIN PRODUCT GOAL
==================================================

A user should be able to do this:

- Open the app
- Search for or add a real book
- Enter title + author + optional series
- Let the app search and build a book profile
- Confirm important context
- Enter where they are in the book
- Optionally paste a short excerpt or describe what just happened
- Generate a spoiler-safe visual story companion
- Reopen later and continue from the same book using saved context

3. PUBLIC SEARCH-POWERED BOOK CONTEXT
==================================================

Implement backend search support for real books.

Goal:
Given title + author (+ optional series), gather public, non-pirated context to help build a usable book profile.

Use public search sources where possible, such as:
- publisher blurbs
- bookstore descriptions
- public book metadata
- author pages
- fan wikis
- series wikis
- spoiler-safe summaries
- public reviews
- discussion sources where helpful

Do NOT:
- scrape or ingest full copyrighted book text unlawfully
- pretend to have access to Kindle/Audible private libraries
- rely on pirated sources
- copy large copyrighted passages

The search result should be distilled into a structured “book context” object, not raw web results.

Create a backend endpoint such as:
POST /api/books/context/search

Request body:
- title
- author
- series
- bookNumber

Response should return a normalized object like:

{
  "book": {
    "title": "",
    "author": "",
    "series": "",
    "bookNumber": null,
    "genre": [],
    "tone": [],
    "publisherBlurb": "",
    "coverImageUrl": null
  },
  "world": {
    "settingSummary": "",
    "visualStyleHints": [],
    "factions": [],
    "locations": [],
    "technology": [],
    "species": [],
    "ships": [],
    "objects": []
  },
  "characters": [
    {
      "name": "",
      "description": "",
      "role": "",
      "visualTraits": [],
      "relationships": []
    }
  ],
  "spoilerSafeContext": {
    "seriesContext": "",
    "nonSpoilerSummary": "",
    "knownThemes": [],
    "cautions": []
  },
  "sources": [
    {
      "title": "",
      "url": "",
      "type": ""
    }
  ]
}

This should be a support layer to generate visuals, not a raw search dump.

- progress info

- whatJustHappened optional

The app should then:
- generate spoiler-safe scene cards
- generate or load matching images
- show the companion quickly
- store the result for reuse

==================================================
7. SCENE GENERATION RULES
==================================================

Scene generation must be spoiler-safe.

Use these principles:
- no major twists
- no deaths unless already explicitly included by the user
- no revealing future outcomes
- focus on atmosphere, environment, mood, visual presence
- describe what can be seen / felt in the moment
- treat the result like a cinematic companion, not a plot summary

Each scene should include:
- title
- narration
- summary
- location
- mood
- characters
- gradient colours
- image prompt
- optional image URL/status

Generate 3 to 5 scenes per chapter/segment.

==================================================
8. IMAGE GENERATION + REUSE

Use the existing idea of server-side image caching/storage.

Important:

If not:

Do not force the user to wait for every image before using the app.

Flow:
- show scene cards first
- use gradient fallback
- load existing images if available
- generate missing images in background
- update the UI progressively
